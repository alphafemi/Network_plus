This  provides a clear, practical overview of the most important **DNS resource record types** you will encounter when managing or troubleshooting DNS. It explains the purpose and format of each record, from basic address mapping to email routing, security, and reverse lookups.



---

### DNS Resource Record Types Cheat Sheet

This guide covers the most common DNS record types, their purpose, and examples.

---

#### Zone Management Records

| Record Type | Name | Purpose | Example |
| :--- | :--- | :--- | :--- |
| **SOA** | **Start of Authority** | Defines the overall zone: domain name, serial number, refresh/retry/expiry timers, and the primary nameserver. | `example.com. 86400 IN SOA ns1.example.com. admin.example.com. (2025010101 7200 3600 1209600 86400)` |
| **NS** | **Name Server** | Identifies the authoritative DNS servers for the domain. | `example.com. 86400 IN NS ns1.example.com.` |

---

#### Address Resolution Records (Forward Lookup)

| Record Type | Name | Purpose | Example |
| :--- | :--- | :--- | :--- |
| **A** | **Address Record (IPv4)** | Maps a hostname to an **IPv4** address (32‑bit). The most common DNS record. | `www.example.com. 300 IN A 192.0.2.10` |
| **AAAA** | **IPv6 Address Record** | Maps a hostname to an **IPv6** address (128‑bit). Often called "quad A." | `www.example.com. 300 IN AAAA 2001:db8::1` |

**Note:** A and AAAA records are the records returned in response to a forward lookup query (name → IP).

---

#### Alias and Mail Records

| Record Type | Name | Purpose | Example |
| :--- | :--- | :--- | :--- |
| **CNAME** | **Canonical Name (Alias)** | Creates an alias for another hostname. The client must resolve the target name separately (which may require an additional query). | `ftp.example.com. 300 IN CNAME mail.example.com.` |
| **MX** | **Mail Exchanger** | Specifies the mail server(s) that accept email for the domain, along with a priority value (lower = higher priority). | `example.com. 300 IN MX 10 mail.example.com.` |

**CNAME Important Note:** Multiple queries may be required. Example:
1. Query for `www.example.com` returns a CNAME to `mail.example.com`.
2. If the client does not have `mail.example.com` cached, it issues a second query for that name's A/AAAA record.

---

#### Security and Verification Records

| Record Type | Name | Purpose | Example |
| :--- | :--- | :--- | :--- |
| **TXT** | **Text Record** | Stores arbitrary human‑readable text. Used for email authentication (SPF, DKIM), domain verification, and other policies. | See examples below |
| **SPF** (within TXT) | **Sender Policy Framework** | Lists the mail servers authorized to send email for the domain. Prevents email spoofing. | `"v=spf1 mx include:mailgun.org ~all"` |
| **DKIM** (within TXT) | **Domain Keys Identified Mail** | Stores the public key used to verify digital signatures on outgoing emails. | `"v=DKIM1; k=rsa; p=MIGfMA0GCSqGSIb3DQEBAQUAA4GNADCBiQKBgQC..."` |

**TXT Record Examples (SPF and DKIM):**

```
# SPF record – authorizes mail servers
professormesser.com. 300 IN TXT "v=spf1 mx include:mailgun.org ~all"

# DKIM record – stores public key for signature verification
mailgun._domainkey.professormesser.com. 300 IN TXT "v=DKIM1; k=rsa; p=MIGfMA0GCSqGSIb3DQEBAQUAA4GNADCBiQKBgQC..."
```

#### Reverse Lookup Record

| Record Type | Name | Purpose | Example |
| :--- | :--- | :--- | :--- |
| **PTR** | **Pointer Record** | Maps an IP address back to a hostname (reverse of an A/AAAA record). Used for reverse DNS lookups. | `15.23.168.192.in-addr.arpa. 300 IN PTR www.example.com.` |

**How Reverse Lookup Works:**

| Step | Action |
| :--- | :--- |
| 1 | Client queries DNS for `15.23.168.192.in-addr.arpa.` (IPv4 reverse domain – `in-addr.arpa`) |
| 2 | DNS server returns a PTR record: `www.example.com.` |

**Reverse DNS Naming Convention:**
- **IPv4:** `[reversed-octets].in-addr.arpa`
- **Example:** IP `192.168.23.15` → `15.23.168.192.in-addr.arpa`
- **IPv6:** `[reversed-hex-nibbles].ip6.arpa`

**PTR Record Examples:**
```
2.23.168.192.in-addr.arpa. 300 IN PTR joe.example.com.
15.23.168.192.in-addr.arpa. 300 IN PTR www.example.com.
17.23.168.192.in-addr.arpa. 300 IN PTR bill.example.com.
```

#### Record Summary Table

| Record | Full Name | Lookup Type | Purpose |
| :--- | :--- | :--- | :--- |
| **A** | Address | Forward (Name → IPv4) | Maps hostname to IPv4 address |
| **AAAA** | IPv6 Address | Forward (Name → IPv6) | Maps hostname to IPv6 address |
| **CNAME** | Canonical Name | Forward (Alias) | Alias for another hostname |
| **MX** | Mail Exchanger | Forward | Email server for the domain |
| **TXT** | Text | Forward | Arbitrary text (SPF, DKIM, verification) |
| **NS** | Name Server | Forward | Authoritative nameservers for the domain |
| **SOA** | Start of Authority | Zone metadata | Zone configuration and timers |
| **PTR** | Pointer | Reverse (IP → Name) | Maps IP address to hostname |

---

#### Key Takeaways

| Takeaway | Explanation |
| :--- | :--- |
| **A and AAAA are the most common records** | They directly answer forward lookup queries (name → IP). |
| **CNAME creates aliases but may require two lookups** | The client resolves the alias, then the target name separately. |
| **MX records control email routing** | Lower priority value = higher preference. |
| **SPF and DKIM use TXT records** | SPF prevents spoofing; DKIM provides email signature verification. |
| **PTR records enable reverse lookups** | Required for reverse DNS (IP → name), often used for mail server validation. |
| **SOA and NS are zone‑level records** | They define the zone's authority and metadata, not individual hosts. |
