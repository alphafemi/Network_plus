This  covers the two primary high-availability redundancy models for network devices: **active-passive** and **active-active**. These configurations are essential for maintaining uptime and availability when a device fails.



---

### Network Redundancy Models Cheat Sheet

This guide covers the two main high-availability configurations for network devices (firewalls, routers, load balancers, etc.): active-passive and active-active.

---

#### Active-Passive (Standby) Redundancy

In an active-passive configuration, two devices are deployed, but only one is actively handling traffic at any given time. The second device remains in standby, ready to take over if the primary fails.

| Aspect | Description |
| :--- | :--- |
| **Active Device** | Handles all production traffic. |
| **Passive Device** | Idle (standby), constantly monitoring the active device's health. |
| **Failover Trigger** | Passive device detects active device has failed (no heartbeat, link down, etc.). |
| **Failover Action** | Passive device takes over as active. |
| **State Synchronization** | Configuration, session tables, routing tables, NAT tables are continuously replicated from active to passive. |

**Requirements for Active-Passive:**
- Identical hardware and software versions.
- Synchronized configurations (changes on active must be copied to passive).
- Synchronized state tables (so existing connections aren't dropped during failover).

**Example: Two Firewalls in Active-Passive**
```
Internet
    |
[FW-A (Active)] --- heartbeat/state sync --- [FW-B (Passive)]
    |
[Router] --- [Switch] --- [Web Server]
```

- Normal operation: Traffic flows through FW-A.
- FW-A fails: FW-B detects loss of heartbeat, transitions to active.
- Traffic now flows through FW-B.

**Pros:**
- Simple to implement.
- Predictable failover behavior.

**Cons:**
- One device sits idle (50% capacity utilization).
- Failover may cause a brief interruption (seconds).

---

#### Active-Active Redundancy

In an active-active configuration, both devices are actively handling traffic simultaneously. This provides higher throughput and automatic load distribution.

| Aspect | Description |
| :--- | :--- |
| **Both Devices** | Active at the same time, sharing the traffic load. |
| **Load Distribution** | Traffic flows are distributed across both devices (e.g., flow A through device 1, flow B through device 2). |
| **Failover Behavior** | If one device fails, the remaining device continues handling all traffic (capacity is reduced but service continues). |
| **Complexity** | Requires careful design of routing, switching, and state synchronization to ensure return traffic goes through the same device (asymmetric routing prevention). |

**Example: Two Firewalls in Active-Active**
```
Internet
    |
[FW-A (Active)] --- state sync --- [FW-B (Active)]
    |                    |
[Router] --- [Switch] --- [Web Server]
```

- Normal: Traffic flows are load-balanced across both firewalls.
- FW-A fails: FW-B handles 100% of traffic (may be degraded but operational).

**Challenges:**
- **Asymmetric Routing:** Traffic from client to server may go through FW-A, but return traffic from server to client may go through FW-B. Firewalls must share session state to allow this.
- **State Synchronization:** Session tables must be synchronized in real time across both devices.
- **Design Complexity:** Requires careful planning of routing, VLANs, and switch configurations.

**Pros:**
- Utilizes both devices (no idle hardware).
- Higher aggregate throughput.
- Automatic load balancing.

**Cons:**
- More complex to configure and troubleshoot.
- Requires robust state synchronization.

---

#### Comparison Table

| Feature | Active-Passive | Active-Active |
| :--- | :--- | :--- |
| **Device Utilization** | 50% (one idle) | 100% (both active) |
| **Throughput** | Limited to single device capacity | Sum of both devices (up to 2x) |
| **Failover Speed** | Seconds (standby must activate) | Instant (remaining device continues) |
| **Complexity** | Low to moderate | High |
| **State Synchronization** | Required | Required (more critical) |
| **Asymmetric Routing Support** | Not applicable | Must be supported |
| **Best For** | Most enterprise environments | High-throughput, maximum utilization scenarios |

---

#### Key Takeaways

| Takeaway | Explanation |
| :--- | :--- |
| **Active-passive is simpler and more common** | One device active, one standby. Failover is straightforward but leaves hardware idle. |
| **Active-active provides higher throughput** | Both devices share the load, utilizing full capacity, but requires more complex design. |
| **State synchronization is critical for both** | Session tables, NAT mappings, and routing tables must be shared between devices for seamless failover. |
| **Active-active must handle asymmetric routing** | Return traffic may take a different path; devices must share state to allow this. |
| **Choose based on needs** | Active-passive for most enterprise firewalls/routers; active-active for high-throughput or maximum utilization requirements. |
