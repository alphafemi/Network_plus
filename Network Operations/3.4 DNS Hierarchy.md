This  provides a comprehensive overview of the **Domain Name System (DNS)** , covering its hierarchical structure, resolution process, caching mechanisms, and security extensions. It explains how DNS translates human-readable names into IP addresses (forward lookup) and vice versa (reverse lookup), along with critical concepts like authoritative vs. non-authoritative servers, TTL, recursive queries, and modern security enhancements like DNSSEC, DoT, and DoH.

### DNS (Domain Name System) Cheat Sheet

This guide covers DNS hierarchy, resolution types, server roles, caching, and security extensions.

---

#### DNS Hierarchy and FQDN

The DNS database is a hierarchical tree structure.

| Level | Example | Description |
| :--- | :--- | :--- |
| **Root** | `.` (dot) | The top of the hierarchy. Managed by 13 root server clusters (over 1,000 physical servers). |
| **Top-Level Domain (TLD)** | `.com`, `.org`, `.net`, `.us`, `.uk` | Generic TLDs (gTLD) and country-code TLDs (ccTLD). |
| **Second-Level Domain** | `professormesser` | Registered domain name. |
| **Subdomain** | `www`, `mail`, `east`, `west` | Optional subdivisions. |
| **Fully Qualified Domain Name (FQDN)** | `www.professormesser.com.` | Complete domain name that specifies its exact position in the DNS tree (includes trailing dot). |

**Example Hierarchy:**
```
. (root)
├── .com
│   └── professormesser
│       ├── www
│       └── mail
└── .org
    └── example
```

#### DNS Server Roles

| Server Type | Description |
| :--- | :--- |
| **Primary (Master) DNS** | Holds the original, writable zone files. All changes (add, modify, delete records) are made here. |
| **Secondary (Slave) DNS** | Receives a read‑only copy of the zone from the primary server (zone transfer). Provides redundancy and load distribution. |
| **Authoritative Server** | The DNS server that holds the definitive records for a specific zone (primary or secondary). |
| **Non‑authoritative Server** | A DNS server that has cached records from an authoritative server. Answers may be stale until TTL expires. |
| **Caching DNS Server** | Stores query results locally to speed up subsequent requests. Most recursive resolvers are caching servers. |

#### Forward Lookup vs. Reverse Lookup

| Lookup Type | Input | Output | Record Type |
| :--- | :--- | :--- | :--- |
| **Forward Lookup** | Name (e.g., `www.example.com`) | IP address | A (IPv4) or AAAA (IPv6) |
| **Reverse Lookup** | IP address | Name (FQDN) | PTR (Pointer record) |

**Example (forward – dig):**
```bash
$ dig www.professormesser.com
;; ANSWER SECTION:
www.professormesser.com. 300 IN A 104.22.72.108
```

**Example (reverse – dig -x):**
```bash
$ dig -x 104.22.72.108
;; ANSWER SECTION:
108.72.22.104.in-addr.arpa. 300 IN PTR cruz.ns.cloudflare.com.
```

---

#### Authoritative vs. Non‑authoritative Answers

| Answer Type | Source | Reliability |
| :--- | :--- | :--- |
| **Authoritative** | Directly from the primary or secondary DNS server that owns the zone. | Always current. |
| **Non‑authoritative** | From a caching DNS server (e.g., Google `8.8.8.8`, Cloudflare `1.1.1.1`). | May be stale; depends on TTL. |

**nslookup Example:**
```
> www.professormesser.com
Server:  9.9.9.9
Non-authoritative answer:
Name:    www.professormesser.com
Addresses:  104.22.72.108
```

#### Time to Live (TTL)

| Aspect | Description |
| :--- | :--- |
| **Definition** | The number of seconds a DNS record can be cached by a non‑authoritative (recursive) server before it must be refreshed from the authoritative source. |
| **Configuration** | Set on the authoritative DNS server (per record or per zone). |
| **Example** | TTL = `300` → cache for 5 minutes. |
| **Trade‑off** | Short TTL = changes propagate quickly, but more queries to authoritative server. Long TTL = less load, but changes take longer to take effect. |

#### Recursive DNS Query Process

When no cached information exists, a recursive resolver follows these steps:

| Step | Query From → To | Information Received |
| :--- | :--- | :--- |
| **1** | Local resolver → Root server | "I don't know `www.example.com`, but here is the `.com` TLD server." |
| **2** | Local resolver → `.com` TLD server | "I don't know `www.example.com`, but here is the authoritative server for `example.com`." |
| **3** | Local resolver → `example.com` authoritative server | "`www.example.com` resolves to `192.0.2.10`." |
| **4** | Local resolver → Client (resolver) | Returns the IP address and caches the result for future queries. |

**Key Point:** The recursive resolver does all the work. The client (your device) only communicates with the local resolver.

#### Local Name Resolution: The Hosts File

The **hosts file** allows local override of DNS resolution without querying a DNS server.

| Aspect | Description |
| :--- | :--- |
| **File Name** | `hosts` (no extension) |
| **Windows Path** | `C:\Windows\System32\drivers\etc\hosts` |
| **Linux/macOS Path** | `/etc/hosts` |
| **Format** | `IP_address hostname` (e.g., `10.1.10.170 www.professormesser.com`) |
| **Use Cases** | Testing (access test server instead of production), overriding incorrect DNS, blocking domains (redirect to `127.0.0.1`). |
| **Limitation** | Not all applications respect the hosts file; some always query DNS directly. |

#### DNS Security Extensions

Traditional DNS sends queries and responses in **cleartext** and provides no authentication of results.

| Security Issue | Description |
| :--- | :--- |
| **No Encryption** | Anyone on the network path can see which domains you are querying. |
| **No Authentication** | Spoofed DNS responses can redirect users to malicious sites (DNS cache poisoning). |

| Solution | Description | Port | Encryption |
| :--- | :--- | :--- | :--- |
| **DNSSEC** (DNS Security Extensions) | Digitally signs DNS responses. Client can verify that the response came from the authoritative server and was not tampered with. | Standard DNS port (53) | **No** (signatures only; payload is still cleartext) |
| **DoT** (DNS over TLS) | Encrypts the entire DNS query and response stream over TLS. Uses a dedicated port. | **TCP 853** | **Yes** (full encryption) |
| **DoH** (DNS over HTTPS) | Encrypts DNS traffic inside standard HTTPS packets. Uses the same port as web traffic. | **TCP 443** (HTTPS) | **Yes** (full encryption) |

**Comparison:**

| Feature | DNSSEC | DoT | DoH |
| :--- | :--- | :--- | :--- |
| **Authentication** | Yes | No (relies on TLS) | No (relies on TLS) |
| **Encryption** | No | Yes | Yes |
| **Port** | 53 | 853 | 443 |
| **Visibility to Firewalls** | Visible (cleartext queries) | Can be detected and blocked (dedicated port) | Blends with web traffic (harder to block) |

---

#### Key Takeaways

| Takeaway | Explanation |
| :--- | :--- |
| **DNS Hierarchically Resolves Names** | Root → TLD → Authoritative → Answer. |
| **Forward = Name → IP; Reverse = IP → Name** | A/AAAA records for forward; PTR records for reverse. |
| **Authoritative = Source of Truth** | Primary holds original zone; secondary is read‑only copy. |
| **Non‑authoritative = Cached Answers** | Recursive resolvers cache responses based on TTL. |
| **TTL Controls Caching Duration** | Shorter TTL = faster updates, more queries. Longer TTL = less load, slower propagation. |
| **Hosts File Overrides DNS Locally** | Useful for testing and overriding bad records. |
| **DNSSEC Provides Authentication** | Digitally signed responses prevent spoofing, but no encryption. |
| **DoT and DoH Provide Encryption** | DoT uses dedicated port 853; DoH hides DNS in HTTPS traffic on port 443. |
