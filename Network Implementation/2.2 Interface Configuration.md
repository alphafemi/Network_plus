### Interface Configuration Cheat Sheet

This guide covers the essential configuration parameters for network interfaces, including troubleshooting mismatches and optimizing performance.

---

#### Ethernet Speed and Duplex

These are physical layer (Layer 1) settings that must match on both ends of a connection.

| Setting | Description | Common Values |
| :--- | :--- | :--- |
| **Speed** | The data rate at which the interface operates. | 10 Mbps, 100 Mbps, 1 Gbps (1000 Mbps), 10 Gbps, 25 Gbps, 40 Gbps, 100 Gbps |
| **Duplex** | Determines whether the interface can send and receive simultaneously. | **Half Duplex:** Can send OR receive at one time (like a walkie-talkie). <br> **Full Duplex:** Can send AND receive simultaneously (like a phone call). |

**Autonegotiation:**
- Most modern interfaces are set to **autonegotiate** both speed and duplex.
- The device and the switch automatically determine the best common settings.

**Troubleshooting Mismatches:**

| Mismatch Type | Result |
| :--- | :--- |
| **Speed Mismatch** | No link. The connection will not come up at all. |
| **Duplex Mismatch** | Link comes up, but performance is poor. <br> **Common scenario:** One side forced to full duplex, the other autonegotiates to half duplex. Results in late collisions, CRC errors, and slow throughput. |

**Best Practice:**
- Use **autonegotiation** whenever possible.
- If manually configuring, ensure **speed and duplex match exactly** on both ends.

#### IP Configuration

Once physical connectivity is established, Layer 3 (IP) settings must be correct for network communication.

| Setting | Description |
| :--- | :--- |
| **IP Address** | Unique identifier for the device on the network. |
| **Subnet Mask** | Defines which portion of the IP address is the network vs. host. |
| **Default Gateway** | The router address used to reach networks outside the local subnet. |
| **DNS Servers** | Resolves domain names to IP addresses. |

**Troubleshooting:**
- Verify all settings match what was assigned by the network administrator.
- Common issues: incorrect default gateway, wrong subnet mask, duplicate IP addresses.
- Use `ipconfig` (Windows) or `ifconfig`/`ip addr` (Linux/macOS) to validate.

#### Link Aggregation (LAG / Port Bonding)

Link aggregation combines multiple physical interfaces into a single logical link, increasing bandwidth and providing redundancy.

| Concept | Description |
| :--- | :--- |
| **LAG (Link Aggregation Group)** | A group of physical ports bundled together to act as one logical interface. |
| **Benefits** | - Increased throughput (e.g., 4 × 1 Gbps = 4 Gbps aggregate) <br> - Redundancy (if one link fails, traffic continues on others) |
| **LACP (Link Aggregation Control Protocol)** | IEEE 802.3ad standard protocol that automates LAG configuration. Switches negotiate and manage the bundle automatically. |
| **Requirement** | Both devices must support LAG/LACP, and all ports in the bundle must have matching configurations (speed, duplex, VLAN). |

**Without LAG:** Multiple connections between switches would create a loop, causing broadcast storms.
**With LAG:** The switch treats the bundle as a single logical link—no loop, no spanning-tree blocking.

#### MTU (Maximum Transmission Unit) and Jumbo Frames

**MTU** is the largest size packet (or frame) that can be transmitted on an interface.

| Concept | Description |
| :--- | :--- |
| **Standard Ethernet MTU** | **1,500 bytes** (the payload size of a standard Ethernet frame). |
| **Path MTU Discovery (PMTUD)** | The automatic process where devices determine the smallest MTU along a path to avoid fragmentation. Can be blocked by firewalls. |

**Jumbo Frames:**
- Frames larger than the standard 1,500 bytes.
- Commonly configured up to **9,000 bytes** (some support up to 9,216 bytes).
- **Benefit:** Fewer frames to process = lower overhead, higher throughput for large data transfers.

| Aspect | Standard MTU | Jumbo Frames |
| :--- | :--- | :--- |
| Frame Size | 1,500 bytes (payload) | Up to 9,000+ bytes |
| Efficiency | More overhead per data transferred | More data per frame, less overhead |
| Requirement | Works everywhere | All devices on the path must support jumbo frames |

**Important Considerations for Jumbo Frames:**
- **All devices in the path** must support jumbo frames (switches, routers, firewalls, endpoints).
- If any device does not support jumbo frames, frames will be **dropped**, not fragmented.
- Often used in **storage networks** (iSCSI, NFS) and data center backbones where all devices are under administrative control.

#### Key Takeaways

| Takeaway | Explanation |
| :--- | :--- |
| **Speed and Duplex Must Match** | Mismatches cause no link (speed) or poor performance (duplex). Autonegotiation is the recommended default. |
| **IP Configuration Must Be Accurate** | Incorrect subnet mask or default gateway can prevent communication even with a valid IP address. |
| **Link Aggregation Increases Throughput** | LAG bundles multiple links into one logical link, increasing capacity and adding redundancy. LACP automates this. |
| **Jumbo Frames Improve Efficiency** | Larger MTUs reduce overhead, but require end-to-end support across all network devices. |
| **Path MTU Discovery Can Fail** | Firewalls blocking ICMP can prevent PMTUD, leading to fragmentation issues or black holes. Manual MTU configuration may be required. |
