### SDN and SD-WAN Cheat Sheet

This guide covers the separation of network functions into planes and how SD-WAN modernizes WAN connectivity for cloud-centric environments.

---

#### The Three Planes of Networking

In traditional networking devices (routers, switches, firewalls), functions are divided into three distinct planes. SDN separates these, allowing them to be virtualized and centrally managed.

| Plane | Also Known As | Function | Examples |
| :--- | :--- | :--- | :--- |
| **Data Plane** | Infrastructure Layer | The **forwarding** function. Does the "heavy lifting" of moving packets from one interface to another. | Forwarding traffic, trunking, encrypting, NAT (Network Address Translation). |
| **Control Plane** | Control Layer | The **decision-making** function. Determines how traffic should be forwarded by building and maintaining tables. | Routing tables, switching tables (MAC address tables), NAT tables, ACLs. |
| **Management Plane** | Application Layer | The **administration** function. Allows network administrators to configure, monitor, and manage the device. | CLI (SSH/console), web-based GUI, SNMP, APIs. |

**Key Takeaway:** SDN virtualizes these functions. The control and management planes can be centrally managed as software, while the data plane remains on the forwarding devices (which can be physical or virtual).

---

#### What is SD-WAN?

**SD-WAN (Software-Defined Wide Area Network)** is an application of SDN principles to wide area networks. It is designed to address the complexity of connecting users to applications that are no longer centralized in a single data center but are distributed across multiple clouds and SaaS providers.

---

#### The Problem: Before SD-WAN

| Traditional WAN | Cloud Era WAN |
| :--- | :--- |
| All services (email, databases, apps) were in a single, centralized data center. | Services are distributed across multiple cloud providers (AWS, Azure, Google, SaaS like Office 365). |
| Remote sites used dedicated WAN links (e.g., MPLS) back to the data center. | A hub-and-spoke model is inefficient—it backhauls traffic to the data center before going to the cloud. |

---

#### Key Features of SD-WAN

| Feature | Description |
| :--- | :--- |
| **Application Awareness** | The network can identify the application being used (e.g., email, voice, database) and forward it to the **optimal destination** (e.g., direct to the closest cloud service instead of backhauling to the data center). |
| **Transport Agnostic** | Can use any combination of transport links: MPLS, broadband internet, fiber, LTE/5G, DSL. SD-WAN intelligently routes traffic over the best available link based on application needs. |
| **Zero-Touch Provisioning (ZTP)** | New devices can be shipped to remote sites, plugged in, and automatically download their configuration from a central controller without any on-site technical expertise. |
| **Central Policy Management** | All network policies (security, routing, QoS) are defined in one central management console and automatically pushed out to all SD-WAN devices across the network. |
| **Direct Cloud Access** | Remote sites can connect directly to cloud-based applications (SaaS, IaaS) without backhauling traffic through a central data center, reducing latency and improving performance. |

---

#### SD-WAN Architecture Visualized

**Before SD-WAN (Traditional Hub-and-Spoke):**
```
Remote Site ----- WAN Link -----> Central Data Center -----> Cloud Services
                                       ^
                                       |
                                 Backhauled traffic
```

**After SD-WAN:**
```
Remote Site ----\                 /-----> Direct to Cloud (SaaS)
                 \               /
                  ---> SD-WAN Router ----> Direct to Cloud (IaaS)
                 /               \
Remote Site ----/                 \-----> Central Data Center (if needed)
```

---

#### Why SD-WAN Matters

| Challenge Addressed | How SD-WAN Solves It |
| :--- | :--- |
| **Cloud Migration** | Enables direct access to cloud services, reducing latency and bandwidth costs associated with backhauling. |
| **Complexity** | Centralized management simplifies configuration and policy deployment across hundreds or thousands of sites. |
| **Cost** | Can use cheaper internet broadband links as primary connections, reserving expensive MPLS for critical traffic or as backup. |
| **Agility** | Zero-touch provisioning allows new sites to be brought online quickly without manual configuration. |
| **Performance** | Application-aware routing ensures that critical traffic (like voice or video) takes the best available path. |
