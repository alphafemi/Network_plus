### Authentication, Authorization, and Accounting (AAA) Cheat Sheet

This guide covers the AAA framework, common authentication protocols (RADIUS, LDAP, SAML, TACACS+), single sign‑on, and multi‑factor authentication.

---

#### The AAA Framework

| Component | Description | Example |
| :--- | :--- | :--- |
| **Identification** | Claiming an identity (usually public information). | Username, email address, employee ID. |
| **Authentication** | Proving you are who you claim to be (private information). | Password, biometric, security token. |
| **Authorization** | Determining what an authenticated user is allowed to do. | Access to files, directories, VPN, or specific network segments. |
| **Accounting (Auditing)** | Logging and tracking authentication events for compliance and investigation. | Login time, logout time, failed login attempts, resources accessed. |

---

#### Common Authentication Protocols

| Protocol | Full Name | Port(s) | Key Characteristics | Use Cases |
| :--- | :--- | :--- | :--- | :--- |
| **RADIUS** | Remote Authentication Dial‑In User Service | UDP 1812 (auth), UDP 1813 (accounting) | Very widely supported; originally for dial‑up, now used for modern networks. | VPN authentication, 802.1X wireless authentication, switch/routers access. |
| **LDAP** | Lightweight Directory Access Protocol | TCP 389 (LDAP), TCP 636 (LDAPS) | Reads/writes to a hierarchical directory (X.500 standard). Provides rich context about users and devices. | Windows Active Directory, Apple OpenDirectory, Novell eDirectory. |
| **SAML** | Security Assertion Markup Language | Web‑based (HTTP/HTTPS) | Open standard for exchanging authentication/authorization data. Uses tokens. Not built for mobile originally. | Web‑based single sign‑on (SSO) between organizations (e.g., logging into a third‑party app with your corporate credentials). |
| **TACACS+** | Terminal Access Controller Access‑Control System Plus | TCP 49 | Originally for ARPANET dial‑up; now considered Cisco‑centric but partially open (standard published in 1993). | Device administration (routers, switches) – separates authentication, authorization, and accounting into separate processes. |

---

#### Single Sign‑On (SSO)

| Aspect | Description |
| :--- | :--- |
| **Definition** | A user authenticates once (e.g., at the start of the workday) and is then automatically granted access to multiple systems and applications without re‑entering credentials. |
| **Typical Duration** | Often configured for a 24‑hour period or a session lifetime. |
| **Benefit** | Reduces password fatigue, improves user experience, centralizes authentication management. |
| **Dependency** | Not all authentication methods support SSO; requires compatible protocols (e.g., SAML, Kerberos). |

---

#### Directory Structure (LDAP / X.500)

LDAP directories are hierarchical, allowing rich context for users and devices.

| Level | Component | Example |
| :--- | :--- | :--- |
| **Root** | Organization / Domain | `widget.com` |
| **Country** | Geographic location | `UK`, `US` |
| **Organization / Department** | Business unit | `Marketing`, `Engineering` |
| **Leaf Objects** | Individual users or devices | `WIDGETWEB` (web server), `jdoe` (user) |

**Attributes Example (for a web server):**
- Common Name (CN): `WIDGETWEB`
- Organizational Unit (OU): `Marketing`
- Organization (O): `Widget`
- Locality (L): `London`
- Domain Component (DC): `widget.com`

**Full Distinguished Name (DN):** `CN=WIDGETWEB, OU=Marketing, O=Widget, L=London, DC=widget, DC=com`

#### Multi‑Factor Authentication (MFA)

MFA requires two or more independent authentication factors to verify a user's identity.

| Factor Category | Description | Examples |
| :--- | :--- | :--- |
| **Something you know** | Knowledge‑based | Password, PIN, security question answer. |
| **Something you have** | Possession‑based | Mobile phone (with authenticator app), hardware token, smart card. |
| **Something you are** | Biometric | Fingerprint, retina scan, facial recognition, hand geometry. |
| **Somewhere you are** | Location‑based | GPS coordinates, IP geolocation, specific WiFi network name (geofencing). |

---

#### Time‑Based One‑Time Password (TOTP)

TOTP is a common implementation of the "something you have" factor.

| Aspect | Description |
| :--- | :--- |
| **Algorithm** | Time‑based One‑Time Password algorithm (RFC 6238). |
| **How It Works** | A secret key is shared between the client (e.g., authenticator app) and the server. The current time (via NTP) is used as an input to generate a pseudo‑random number that changes every 30 seconds. |
| **Why It's Secure** | The number is not truly random; both sides compute the same expected value based on the secret key and the time. |
| **Examples** | Google Authenticator, Microsoft Authenticator, Authy, many others. |
| **Typical Flow** | User enters username + password + the 6‑digit TOTP code from their phone. |

**Note:** TOTP relies on accurate time synchronization (NTP) between the client and the server.

#### Authentication Flow Examples

**RADIUS Flow (VPN Authentication):**
```
Laptop (Client) → VPN Concentrator → RADIUS Server → Authentication Database
                                              ↓
                              Success/Failure response back to client
```

**SAML Flow (Web SSO):**
```
User (Browser) → Resource Server → Redirect to Authorization Server → User authenticates
                                                                           ↓
                                            Authorization Server returns token to browser
                                                                           ↓
                                            Browser presents token to Resource Server
                                                                           ↓
                                                               Access granted
```

---

#### Key Takeaways

| Takeaway | Explanation |
| :--- | :--- |
| **AAA = Authentication, Authorization, Accounting** | Three separate functions that together control access and track activity. |
| **RADIUS is ubiquitous** | Supported on virtually all network devices (VPN, wireless, switches). |
| **LDAP provides directory context** | Hierarchical structure (X.500) adds rich attributes to users and devices. |
| **SAML enables web SSO** | Open standard for exchanging authentication data; common for cloud applications. |
| **TACACS+ is device‑administration focused** | Often used for router/switch login, with separate authentication/authorization/accounting. |
| **MFA adds layers of security** | Combining "something you know" + "something you have" (e.g., password + TOTP) dramatically reduces credential theft risk. |
| **TOTP uses time + secret key** | Code changes every 30 seconds; requires accurate time via NTP. |
