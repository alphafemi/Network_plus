This  provides a clear and detailed explanation of how **MAC addresses** work, how switches build their **MAC address tables** (CAM tables), and how an attacker can exploit this process through a **MAC flooding** attack. It also explains the switch's normal behavior when a destination MAC is unknown and how port security can mitigate this attack.



---

### MAC Addresses and Switch Operation Cheat Sheet

This guide covers MAC address structure, how switches learn and forward traffic, and the MAC flooding attack.

---

#### MAC Address Structure

| Aspect | Description |
| :--- | :--- |
| **Full Name** | Media Access Control address (hardware address, physical address, burned‑in address). |
| **Length** | 48 bits (6 bytes). |
| **Format** | Displayed as hexadecimal, often separated by colons, hyphens, or periods. |
| **Example** | `8C:2D:AA:4B:98:A7` |
| **Storage** | Stored in ROM (read‑only memory) on the network interface card (NIC). |

**Breakdown:**

| Field | Length | Description |
| :--- | :--- | :--- |
| **OUI (Organizationally Unique Identifier)** | First 3 bytes (24 bits) | Identifies the manufacturer of the NIC (e.g., Intel, Cisco, Broadcom). |
| **NIC‑Specific Value** | Last 3 bytes (24 bits) | Effectively a serial number unique to that specific adapter card. |

**Key Point:** The combination of OUI + NIC‑specific value should be globally unique.

---

#### Switch MAC Address Table (CAM Table)

| Aspect | Description |
| :--- | :--- |
| **Purpose** | The switch uses this table to determine which port to forward traffic to for a given destination MAC address. |
| **How It Learns** | The switch examines the **source MAC address** of every frame it receives. If the source MAC is not in the table, it adds an entry: MAC address + ingress port. |
| **Aging** | Entries are temporary. If a MAC address is not seen for a period (typically ~5 minutes), it is removed from the table. |
| **Table Size** | Limited by switch hardware (varies by model; enterprise switches can store tens of thousands of MAC addresses). |

---

#### Normal Switch Forwarding Behavior

| Scenario | Switch Action |
| :--- | :--- |
| **Destination MAC known** | Switch forwards the frame **only** out the port associated with that MAC address (unicast forwarding). |
| **Destination MAC unknown** | Switch forwards the frame out **all ports** except the port it was received on (flooding). This is the normal fail‑safe behavior to ensure delivery. |

**Example (Known Destination):**
- Sam (`MAC: 1000:1111:1111`) sends to SGC Server (`MAC: 1000:5555:5555`).
- Switch looks up destination MAC in its table, finds port F0/5.
- Frame is sent only out F0/5. Other devices on the switch do **not** receive the frame.

**Example (Unknown Destination – Flooding):**
- If the destination MAC is not in the table, the switch sends the frame out **all** ports (except the ingress port), acting like a hub.

---

#### MAC Flooding Attack

| Aspect | Description |
| :--- | :--- |
| **Attack Method** | Attacker sends thousands or millions of frames, each with a **different source MAC address**. |
| **Effect** | The switch's MAC address table fills up (overflows) because it must learn every new source MAC. |
| **Once Table is Full** | The switch can no longer learn new addresses. It falls back to flooding behavior for **all** frames (since destinations are effectively "unknown"). |
| **Result** | The switch becomes a hub – every frame is sent to every port. The attacker can capture traffic not originally destined for them. |

**Attack Diagram:**
```
[Attacker] → sends frames with random source MACs → [Switch] → MAC table fills
                                                              ↓
                                         Switch now floods all traffic to all ports
                                                              ↓
                                         Attacker can see traffic between other devices
```

**Why This Works:**
- Switches have limited CAM table space.
- Flooding is the normal, safe behavior when a destination MAC is unknown.
- The attacker exploits this design to force the switch into broadcast‑like mode.

---

#### Mitigation: Port Security

| Feature | Description |
| :--- | :--- |
| **Port Security** | A switch feature that limits the number of MAC addresses allowed on a given port. |
| **How It Helps** | Prevents an attacker from flooding the switch with thousands of different source MACs from a single port. |
| **Typical Configuration** | Set a maximum number of MAC addresses per port (e.g., 1 for an end‑user workstation, 10 for a printer, etc.). |
| **Violation Actions** | Shutdown the port, restrict (drop offending frames but keep port up), or protect (drop frames without notification). |

**Example Cisco Port Security:**
```
interface GigabitEthernet0/1
 switchport mode access
 switchport port-security
 switchport port-security maximum 1
 switchport port-security violation shutdown
```

**Limitations:**
- Port security reduces the risk but does not eliminate it entirely (a distributed attack across many ports could still fill the table).
- Modern enterprise switches have much larger MAC tables than older switches, making flooding more difficult.

---

#### Key Takeaways

| Takeaway | Explanation |
| :--- | :--- |
| **MAC addresses are 48‑bit hardware addresses** | First 24 bits = OUI (manufacturer); last 24 bits = NIC‑specific (serial). |
| **Switches learn MAC addresses from source frames** | Builds a MAC address table (CAM table) to forward traffic efficiently. |
| **Unknown destination = flooding** | Switch sends the frame out all ports as a fail‑safe to ensure delivery. |
| **MAC flooding attack fills the CAM table** | Attacker sends frames with many random source MACs to exhaust table space. |
| **Full table = switch becomes a hub** | All frames are flooded to all ports; attacker can eavesdrop on other traffic. |
| **Port security limits MACs per port** | Prevents a single attacker from flooding the table. |
