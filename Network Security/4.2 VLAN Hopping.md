This  provides a clear, technical explanation of two common **VLAN hopping** attack techniques: **switch spoofing** and **double tagging**. It also explains how to prevent these attacks through proper switch configuration.



---

### VLAN Hopping Attacks Cheat Sheet

This guide covers how attackers can move between VLANs without a router and the mitigations to prevent these attacks.

---

#### What is VLAN Hopping?

| Aspect | Description |
| :--- | :--- |
| **Definition** | An attack that allows a device on one VLAN to communicate with a device on a different VLAN without going through a router (Layer 3 device). |
| **Why It Works** | Takes advantage of misconfigurations in switch trunking protocols or native VLAN settings. |
| **Risk** | Bypasses the normal segmentation that VLANs are designed to provide. |

---

#### Attack 1: Switch Spoofing

| Aspect | Description |
| :--- | :--- |
| **How It Works** | Attacker pretends to be a switch. The switch autonegotiates a trunk link with the attacker's device. |
| **Key Protocol** | DTP (Dynamic Trunking Protocol) – a Cisco protocol that automatically negotiates trunking between switches. |
| **Attacker's Goal** | Once a trunk is established, the attacker can send tagged frames to any VLAN that is allowed on that trunk port. |
| **Result** | Attacker can access any VLAN that the switch would normally allow over a trunk. |

**Why Switch Spoofing Works:**
- Many switches default to "dynamic auto" or "dynamic desirable" on ports, meaning they will negotiate trunking.
- If an attacker sends DTP frames, the switch may turn the port into a trunk.
- The attacker can then send 802.1Q‑tagged frames for any VLAN.

**Mitigation (Switch Spoofing):**

| Action | Description |
| :--- | :--- |
| **Disable DTP** | Configure switch ports as **static access ports** (not dynamic). |
| **Manual Trunk Configuration** | If a trunk is needed, configure it manually with the `switchport mode trunk` command. Do not use `dynamic auto` or `dynamic desirable`. |
| **Limit Allowed VLANs** | On trunk ports, explicitly define which VLANs are allowed (`switchport trunk allowed vlan X,Y,Z`). Do not allow all VLANs (default). |

**Example Cisco Configuration (Secure):**
```
interface GigabitEthernet0/1
 switchport mode access
 no switchport trunk encapsulation
!
interface GigabitEthernet0/24   (trunk to another switch)
 switchport mode trunk
 switchport trunk allowed vlan 10,20,30
```

#### Attack 2: Double Tagging (802.1Q‑in‑802.1Q)

| Aspect | Description |
| :--- | :--- |
| **How It Works** | Attacker crafts a frame with **two VLAN tags** (double encapsulation). The first tag matches the native VLAN, the second tag is the target VLAN. |
| **Key Requirement** | Attacker must be on the **native VLAN** of the trunk port. |
| **Traffic Direction** | **One‑way only** – responses cannot return because the return path is blocked. Useful for DoS or sending data to a victim (not bidirectional communication). |

**Double Tagging Step‑by‑Step:**

| Step | Description |
| :--- | :--- |
| **1. Frame Construction** | Attacker creates a frame with two 802.1Q tags: outer tag = native VLAN (e.g., VLAN 10), inner tag = target VLAN (e.g., VLAN 20). |
| **2. Transmission** | Attacker sends the double‑tagged frame to the first switch. |
| **3. First Switch (Ingress)** | The first switch sees the outer tag (VLAN 10). Because this is the native VLAN, the switch strips the outer tag and forwards the frame (still containing the inner VLAN 20 tag) across the trunk. |
| **4. Second Switch (Egress)** | The second switch sees the remaining tag (VLAN 20). It strips that tag and delivers the payload to VLAN 20. |
| **5. Victim Receives** | The frame arrives on the victim's port (on VLAN 20), even though the attacker is on VLAN 10. |


**Why Double Tagging Works:**
- Switches strip the **outer tag** when the frame enters on the native VLAN.
- They do not check for additional tags beyond the first.
- The inner tag is then processed by the next switch.

**Mitigation (Double Tagging):**

| Action | Description |
| :--- | :--- |
| **Change the Native VLAN** | Do not use the default native VLAN (usually VLAN 1). Change it to an unused, distinct VLAN ID. |
| **Force Tagging on Native VLAN** | Configure the switch to tag the native VLAN on trunk ports (e.g., `switchport trunk native vlan tag` on some platforms). This prevents untagged frames from being interpreted as native VLAN traffic. |
| **Best Practice** | Explicitly set the native VLAN to a dedicated VLAN that is not used for user traffic. |

VLAN Trunking
What is a Trunk?

Connection between two switches
Carries traffic from multiple VLANs simultaneously
Uses VLAN tags (802.1Q) to identify which VLAN each frame belongs to

#### Summary of Mitigations

| Attack | Mitigation |
| :--- | :--- |
| **Switch Spoofing** | Disable DTP; manually configure trunk ports; limit allowed VLANs on trunks. |
| **Double Tagging** | Change native VLAN from default (VLAN 1); force tagging of native VLAN traffic. |

---

#### Key Takeaways

| Takeaway | Explanation |
| :--- | :--- |
| **VLAN hopping bypasses segmentation** | Attackers can reach VLANs they should not have access to. |
| **Switch spoofing exploits DTP autonegotiation** | Disable DTP and manually configure trunk ports to prevent this. |
| **Double tagging exploits native VLAN misconfiguration** | Change the native VLAN away from the default (VLAN 1). |
| **Double tagging is one‑way only** | Not useful for interactive communication, but effective for DoS or unidirectional data injection. |
| **Manual configuration is more secure** | Avoid "dynamic" or "auto" settings for trunk negotiation. |
