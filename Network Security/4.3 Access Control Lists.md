### Access Control Lists (ACLs) and Firewall Policies Cheat Sheet

This guide covers how to filter traffic based on IP addresses, ports, URLs, content, and security zones.

---

#### Access Control List (ACL) Basics

| Aspect | Description |
| :--- | :--- |
| **Definition** | A list of rules that define which traffic is allowed or denied through a network device (router, firewall, operating system). |
| **Match Criteria** | Source IP, destination IP, port number, protocol (TCP/UDP/ICMP), time of day, application, username. |
| **Evaluation Order** | Rules are evaluated **top‑to‑bottom**. The first rule that matches the traffic determines the action (allow or deny). |
| **Implicit Deny** | If no rule matches, the traffic is **denied** by default (an invisible "deny all" at the end of the list). |

**Best Practice:** Place more **specific** rules at the top of the ACL and more **general** rules toward the bottom.

---

#### Firewall Rule Example (Top‑to‑Bottom Evaluation)

| Rule # | Source IP | Source Port | Destination Port | Protocol | Action | Purpose |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| 1 | Any | Any | 22 (SSH) | TCP | Allow | Allow SSH access from anywhere |
| 2 | Any | Any | 80 (HTTP) | TCP | Allow | Allow web traffic |
| 3 | Any | Any | 443 (HTTPS) | TCP | Allow | Allow secure web traffic |
| 4 | Any | Any | 3389 (RDP) | TCP | Allow | Allow Remote Desktop |
| 5 | Any | Any (local) | 53 (DNS) | UDP | Allow | Allow outbound DNS queries |
| 6 | Any | Any (local) | 123 (NTP) | UDP | Allow | Allow outbound time sync |
| 7 | Any | Any | ICMP | ICMP | Deny | Block ping (ICMP echo request) |
| ... | ... | ... | ... | ... | ... | ... |
| (Implicit) | Any | Any | Any | Any | Deny | Deny everything else |

**Note:** Rule 7 blocks ICMP (ping), while rules 5 and 6 allow outbound DNS and NTP (note that destination port is specified, not source port).

#### URL Filtering (Content Filtering)

| Aspect | Description |
| :--- | :--- |
| **What It Filters** | Uniform Resource Locators (URLs) or Uniform Resource Identifiers (URIs). |
| **How It's Implemented** | Allow list (only these sites allowed) or block list (these sites blocked). |
| **Categorization** | Instead of listing individual URLs, most solutions use **categories** (e.g., auction sites, hacking, travel, recreation, NSFW content). |
| **Circumvention Risk** | Users may try proxies, VPNs, or URL shorteners to bypass filtering. |
| **Best Practice** | Combine URL filtering with firewall rules (e.g., allow only port 80/443 outbound, require authentication). |
| **Common Location** | Built into next‑generation firewalls (NGFW). |

---

#### Other Content Filtering

| Type | Description | Example |
| :--- | :--- | :--- |
| **Data Loss Prevention (DLP)** | Detect and block sensitive data leaving the network (financial details, internal documents, PII). | Prevent credit card numbers from being emailed externally. |
| **Antivirus / Antimalware** | Filter malicious content (executables, scripts, exploits). | Block a known ransomware binary from being downloaded. |
| **Parental Controls** | Block inappropriate content (violence, adult, gambling). | Home router filtering for children. |

#### Screened Subnet (DMZ)

| Aspect | Description |
| :--- | :--- |
| **Also Known As** | DMZ (Demilitarized Zone). |
| **Purpose** | A separate network segment for public‑facing services (web servers, email servers, FTP). |
| **Security Benefit** | Public traffic goes to the screened subnet, not the internal network. If a public server is compromised, the attacker still cannot reach internal systems directly. |
| **Typical Architecture** | `Internet → Firewall → Screened Subnet (DMZ) → (Separate Firewall or Rule) → Internal Network` |

**Access Rules (Typical):**
- Internet → Screened Subnet: allowed only to specific services (HTTP, HTTPS, SMTP).
- Screened Subnet → Internet: allowed (e.g., for updates).
- Screened Subnet → Internal Network: blocked or highly restricted.
- Internal Network → Screened Subnet: allowed for management.

---

#### Security Zones (Zone‑Based Firewalls)

| Aspect | Description |
| :--- | :--- |
| **Definition** | Logical grouping of interfaces (or networks) into named zones (e.g., Trusted, Untrusted, DMZ, Database). |
| **Benefits** | Simplifies rule creation – you write rules based on zones, not individual IP addresses. |
| **Examples** | Trusted (inside), Untrusted (internet), Screened Subnet (DMZ), Server Zone, Database Zone. |

**Example Zone‑Based Rules:**
- `Trusted → Untrusted` : Allow all (internal users to internet)
- `Untrusted → Screened Subnet` : Allow only HTTP/HTTPS (public to web server)
- `Untrusted → Trusted` : Deny all
- `Screened Subnet → Database Zone` : Deny all

**Zone Diagram Concept:**
```
[Internet] (Untrusted Zone)
     ↓
[Firewall]
     ↓
[Screened Subnet Zone] (Public web/mail servers)
     ↓ (separate firewall or rule)
[Internal Network Zone] (Trusted)
```

#### ACL / Firewall Rule Structure Summary

| Component | Description | Example Values |
| :--- | :--- | :--- |
| **Source** | Where the traffic originates | IP address, subnet, zone (Trusted), "Any" |
| **Destination** | Where the traffic is going | IP address, subnet, zone (Untrusted), "Any" |
| **Service / Port** | Protocol and port number | TCP/80, UDP/53, ICMP |
| **Action** | What to do when the rule matches | Allow, Deny, Reject |
| **Order** | Rule number (lower number evaluated first) | 1, 2, 3... |

---

#### Key Takeaways

| Takeaway | Explanation |
| :--- | :--- |
| **ACLs are processed top‑to‑bottom** | The first matching rule determines the action. |
| **Implicit deny is at the end** | Traffic not matching any rule is denied automatically. |
| **URL filtering uses categories** | Block entire classes of websites (hacking, gambling, social media) rather than individual URLs. |
| **Screened subnet (DMZ) isolates public services** | Keeps public traffic away from your internal network. |
| **Security zones simplify firewall rules** | Write rules based on zone names (Trusted → Untrusted) instead of IP addresses. |
| **Combine multiple filtering methods** | Use ACLs, URL filtering, content filtering, and zones together for defense in depth. |
