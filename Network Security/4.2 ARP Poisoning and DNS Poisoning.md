This  provides a clear and practical explanation of **spoofing attacks**, specifically focusing on **ARP poisoning** (also called ARP spoofing) and **DNS poisoning** (DNS spoofing). It explains how these attacks work, why they are effective, and how they enable on‑path (man‑in‑the‑middle) attacks.

### Spoofing Attacks: ARP Poisoning and DNS Poisoning Cheat Sheet

This guide covers how attackers impersonate legitimate devices or services using spoofing techniques, with detailed walkthroughs of ARP and DNS poisoning.

---

#### What is Spoofing?

| Aspect | Description |
| :--- | :--- |
| **Definition** | When one person or device pretends to be another person or device. |
| **Examples** | Fake web server, fake DNS server, forged email sender address, fake caller ID. |
| **Purpose** | Gain access that would normally be denied; enable on‑path (man‑in‑the‑middle) attacks; redirect victims to malicious sites. |

---

#### ARP Poisoning (ARP Spoofing)

ARP poisoning exploits the lack of authentication in the Address Resolution Protocol.

| Aspect | Description |
| :--- | :--- |
| **What ARP Does** | Resolves an IP address to a MAC address on a local network. |
| **Normal ARP Process** | Device sends a broadcast: "Who has IP X?" The owner of that IP responds with its MAC address. The requesting device caches the result. |
| **Vulnerability** | ARP has no authentication. Devices accept unsolicited ARP replies (gratuitous ARP) and update their cache. |
| **Attack** | Attacker sends spoofed ARP replies, claiming to be the IP address of another device (e.g., the default gateway). |

**ARP Poisoning Step‑by‑Step (Example):**

| Step | Description |
| :--- | :--- |
| **1. Normal State** | Device (`192.168.1.9`) has ARP cache entry: `192.168.1.1` → MAC address of legitimate router. |
| **2. Attacker Sends Spoofed ARP** | Attacker (`192.168.1.14`) sends an unsolicited ARP reply to the victim, claiming: "I am `192.168.1.1` and my MAC address is [attacker's MAC]." |
| **3. Victim Updates Cache** | The victim accepts the spoofed reply and updates its ARP cache. Now `192.168.1.1` maps to the attacker's MAC. |
| **4. Traffic Interception** | The victim sends all traffic destined for the router to the attacker instead. |
| **5. Forwarding (Optional)** | The attacker forwards the traffic to the real router, maintaining the connection and staying invisible. |

**Result:** The attacker is now an on‑path (man‑in‑the‑middle) between the victim and the router, able to monitor, modify, or drop traffic.

**Defenses:**
| Defense | Description |
| :--- | :--- |
| **Static ARP entries** | Manually configure ARP mappings for critical devices (not scalable). |
| **Dynamic ARP Inspection (DAI)** | Switch feature that validates ARP packets against a trusted database (DHCP snooping binding table). |
| **Port Security** | Limits which MAC addresses can communicate on a port. |
| **Network Segmentation** | Limit ARP poisoning scope by reducing broadcast domain size. |

---

#### DNS Poisoning (DNS Spoofing)

DNS poisoning modifies DNS resolution so victims are redirected to malicious sites.

| Aspect | Description |
| :--- | :--- |
| **What DNS Does** | Resolves domain names to IP addresses. |
| **Attack Vectors** | Modify the DNS server's zone file (direct compromise), spoof DNS responses (on‑path), or modify the client's **hosts file**. |
| **Hosts File Priority** | The local hosts file has higher priority than DNS queries. If an entry exists in the hosts file, the system uses it without asking a DNS server. |

**DNS Poisoning Step‑by‑Step (Spoofed Responses via On‑Path):**

| Step | Description |
| :--- | :--- |
| **1. Legitimate Request** | Victim sends a DNS query for `professormesser.com` to a legitimate DNS server. |
| **2. Attacker Intercepts** | Using ARP poisoning or another on‑path technique, the attacker sees the request. |
| **3. Attacker Sends Spoofed Response** | The attacker quickly sends a forged DNS response (before the real DNS server responds), claiming `professormesser.com` resolves to the attacker's IP address. |
| **4. Victim Caches Poisoned Entry** | The victim accepts the first response it receives and caches the false mapping. |
| **5. Redirection** | When the victim visits `professormesser.com`, they are sent to the attacker's malicious web server. |

**DNS Poisoning via Direct Server Compromise:**

| Step | Description |
| :--- | :--- |
| **1. Attacker Compromises DNS Server** | Attacker gains access to the DNS server and changes the A record for `professormesser.com` to the attacker's IP address. |
| **2. Subsequent Queries Poisoned** | All future queries to that DNS server return the false IP address. |
| **3. Multiple Victims** | Every client using that compromised DNS server is redirected to the malicious site. |

**DNS Poisoning via Hosts File Modification:**

| Step | Description |
| :--- | :--- |
| **1. Attacker Gains Access** | Attacker modifies the victim's `hosts` file (e.g., via malware or physical access). |
| **2. Entry Added** | `192.168.1.14 www.professormesser.com` is added to the file. |
| **3. Permanent Redirection** | Every time the victim visits the site, they are redirected to the attacker's IP, regardless of DNS settings. |

**Defenses:**

| Defense | Description |
| :--- | :--- |
| **DNSSEC (DNS Security Extensions)** | Digitally signs DNS responses, preventing spoofing and tampering. |
| **DoT / DoH** | Encrypts DNS traffic, preventing on‑path attackers from seeing or modifying queries/responses. |
| **Cache Poisoning Countermeasures** | Modern DNS resolvers use random source ports and transaction IDs to make spoofing difficult. |
| **Hosts File Monitoring** | Monitor hosts file for unauthorized changes (using file integrity monitoring). |
| **Use Authoritative DNS** | For critical domains, ensure DNS servers are hardened and kept up‑to‑date. |

---

#### ARP Poisoning vs. DNS Poisoning Comparison

| Aspect | ARP Poisoning | DNS Poisoning |
| :--- | :--- | :--- |
| **Target** | Local network (broadcast domain) | DNS queries (local or global) |
| **What is Spoofed** | IP‑to‑MAC mapping | Domain‑to‑IP mapping |
| **Scope** | Devices on the same subnet | Any client using the poisoned DNS server or on‑path attacker |
| **Persistence** | Temporary (cache entries age out) | Can be persistent (hosts file, compromised server) |
| **Typical Goal** | On‑path (man‑in‑the‑middle) attack | Redirection to malicious sites |
| **Defense** | Dynamic ARP Inspection (DAI), static ARP | DNSSEC, DoH/DoT, hosts file monitoring |

---

#### Key Takeaways

| Takeaway | Explanation |
| :--- | :--- |
| **Spoofing = pretending to be someone else** | Used to gain unauthorized access or redirect victims. |
| **ARP poisoning exploits no authentication in ARP** | Attacker sends fake ARP replies to become a man‑in‑the‑middle. |
| **DNS poisoning corrupts name resolution** | Victims are redirected to attacker‑controlled IP addresses. |
| **Hosts file overrides DNS** | Modifying the hosts file is a simple but effective form of DNS poisoning. |
| **On‑path attacks are common enablers** | ARP poisoning is often used to position the attacker to then perform DNS poisoning. |
| **Defenses exist but must be configured** | DAI, DNSSEC, DoT/DoH, and network segmentation all help prevent spoofing attacks. |
